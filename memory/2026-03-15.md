Ramon tried to file the first reimbursement case (DAM-CAN2P, row 2) in Seller Central. The new Damaged Warehouse portal asked for a TRID and rejected `20075449481030`.

### Root Cause (NOT a format issue)

- The reference IDs we had ARE the correct format
- The portal rejected it because that specific unit was **already reconciled** (damaged then found/recovered same day)

### Two Major Bugs Found in `reimbursement_audit.py`

**Bug 1: Not netting found/recovered entries against damage/lost entries**
- Inventory Ledger has F (found) and N (found) entries that offset M (misplaced/lost) and Q (damaged) entries
- Example: CAN2P had -1 damaged + +1 found = net 0, but audit showed as 1 unit to claim
- Fix: Added per-SKU (lost) and per-FNSKU (damaged) netting. Only surface cases where net is negative.

**Bug 2: Counting disposition transfers as "lost"**
- `reason='Q'` with `disposition='SELLABLE'` means Amazon transferred from sellable to defective
- These are NOT actual losses — they're CPC compliance reclassifications or damage assessments
- The CANSET "76 lost units" ($623) and GTV-TH "56 lost units" ($465) were actually CPC compliance removals Ramon requested
- Fix: Lost-in-warehouse now only uses `reason='M'` (truly misplaced). Q entries with SELLABLE disposition are excluded.

## FBA Reimbursement Workflow — Walkthrough with Ramon

### Key Learnings
1. **SP-API `QuantityReceived` field lags behind Amazon's actual reconciliation.** The Shipping Queue UI is the source of truth for shipment discrepancies.
2. **Always check IDR (Inventory Defect and Reimbursement) portal first** for warehouse/customer return issues. If it's not "Eligible for Claim" in IDR, there's nothing to file.
3. **Broken 6-pack theory:** If a CAN6P breaks during receiving, Amazon may scan individual CAN2P decks (1 CAN6P = 3 CAN2P). Look for offsetting +/- across related SKUs in the same shipment before flagging.

### Reimbursement Process Rules (Ramon confirmed)
- **Warehouse/Customer issues:** Always check IDR first. Only alert if IDR shows "Eligible for Claim."
- **Shipment discrepancies:** Wait for CLOSED status. Verify via browser (not just API). Ramon will walk through filing process next time one comes up.
