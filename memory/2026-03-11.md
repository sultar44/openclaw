## Major Decision: Replace Airbyte with Direct SP-API Pipeline (~10:00 AM - 11:30 AM)

### Context
After discussing Airbyte's partial failures, Ramon decided to investigate whether we could replace Airbyte entirely. Key insight from his ecom/OpenClaw group: hitting Amazon with too many API calls at once causes timeouts.

### New Architecture (Airbyte Replacement)
Replacing Airbyte with 3 tiers of data collection, all going to BQ:

**Tier 1: Amazon Ads API — Email Delivery (5 reports)**
- Campaigns, Keywords, Targets, ASIN Keywords, ASIN Targets
- Scheduled via Ads API to be delivered as emails
- Separate email-to-BQ pipeline processes them (first run today)

**Tier 2: SP-API Scheduled Reports — Hourly Polling (14 schedules)**
- Set up 16 schedules (8 report types × 2 marketplaces US+CA), removed 2 S&T schedules later
- Reports: FBA Inventory, Orders, Returns, Reimbursements, Removals, Seller Feedback, SQP (weekly)
- Poller: `scheduled_report_poller.py` checks hourly for completed reports
- Settlement reports (`GET_V2_SETTLEMENT_REPORT_DATA_FLAT_FILE`) added to poller — auto-detected when Amazon generates them (~every 2 weeks), loads to `sp_settlement`

**Tier 3: On-Demand Reports — Staggered 3 per day (gentle approach)**
- 7:00 AM: `GET_FBA_INVENTORY_PLANNING_DATA` (US + CA, 60s gap between marketplaces)
- 8:00 AM: `GET_MERCHANT_LISTINGS_ALL_DATA` (US only)
- 9:00 AM: `GET_SALES_AND_TRAFFIC_REPORT` (3-day lookback, US + CA)
- Script: `collectors/on_demand_report.py` — gentle, sequential, no parallel calls
- Sales & Traffic CANNOT be scheduled because traffic data doesn't settle for ~72 hours

**Tier 4: Financial Events API — Daily 5:30 AM**
- `ListFinancialEventGroups` + `ListFinancialEvents` (Finances API endpoints, not reports)
- Script: `collectors/financial_events.py`
- 30-day lookback daily, loads to `sp_financial_event_groups` + `sp_financial_events`
- QBO prep — Ramon wants to start QBO integration within 1 month
- Initial backfill: 90 days, 30 groups, 10,124 events loaded to BQ
